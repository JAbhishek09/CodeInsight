## CRITICAL

---

### BUG-001 · CRITICAL · Security
**Missing `@google/generative-ai` in backend/package.json**

- **File:** `backend/package.json`
- **Root Cause:** `aiService.js` imports `@google/generative-ai` but it is not listed as a dependency. The server starts successfully only if node_modules was previously populated (e.g. by a manual `npm install @google/generative-ai`). On a fresh deploy (`npm ci` or `npm install` from package.json), the import fails at runtime with `Cannot find package '@google/generative-ai'`, crashing the entire backend.
- **Severity:** CRITICAL — fresh deployment kills the backend entirely
- **Fix:** Add `"@google/generative-ai": "^0.21.0"` to `dependencies`
- **Status:** ✅ FIXED (see package.json update below)

---

### BUG-002 · CRITICAL · Security
**CORS wide open — all origins accepted**

- **File:** `backend/server.js` line 22
- **Root cause:** `app.use(cors())` with zero options accepts requests from any origin. In production this means any website can call your authenticated API using the user's stored credentials (CSRF-style attack via credentialed cross-origin request).
- **Fix:** Restrict to `CLIENT_URL` env var
- **Status:** ✅ FIXED

---

### BUG-003 · CRITICAL · Auth
**JWT expiry hardcoded to 30 days — ignores `JWT_EXPIRES_IN` env var**

- **File:** `backend/utils/generateToken.js` line 10
- **Root cause:** `expiresIn: '30d'` is hardcoded. The `.env.example` documents `JWT_EXPIRES_IN=7d` but it is never read. 30-day non-revocable JWTs are a significant security risk.
- **Fix:** Use `process.env.JWT_EXPIRES_IN || '7d'`
- **Status:** ✅ FIXED

---

### BUG-004 · CRITICAL · Data Integrity
**`sync.controller` uses `$push` without deduplication — duplicates submissions on every sync**

- **File:** `backend/controllers/sync.controller.js` lines 29–40, 54–65
- **Root cause:** Every sync call does `$push: { submissions: {...} }` unconditionally. If the user syncs twice, every submission is duplicated in the array. After 10 syncs, each problem has 10 copies of every submission.
- **Fix:** Use `$addToSet` by submissionId, or check existence first. Because `$addToSet` doesn't work on sub-documents with multiple fields, the correct approach is: skip the push if `submissionId` already exists in the array.
- **Status:** ✅ FIXED

---

### BUG-005 · CRITICAL · Missing Auth on Sync & Analytics Routes
**`/api/sync` and `/api/analytics` are mounted BEFORE `protect` in server.js — but the middleware is NOT applied in server.js**

- **File:** `backend/routes/sync.routes.js`, `backend/routes/analytics.routes.js`
- **Root cause:** Both route files have comments saying "protect middleware applied in server.js" but server.js never applies it. The sync and analytics endpoints are completely unauthenticated. Anyone can trigger a sync for any user.
- **Fix:** Add `protect` middleware directly in the route files
- **Status:** ✅ FIXED

## HIGH

### BUG-006 · HIGH · Performance
**`sync.controller` N+1 DB problem — one write per submission**

- **File:** `backend/controllers/sync.controller.js` lines 25–50
- **Root cause:** For 100 CF submissions, this fires 100 sequential `findOneAndUpdate` calls. With 300ms average MongoDB Atlas round-trip this blocks for ~30 seconds.
- **Fix:** Use `bulkWrite` with `updateOne` + `upsert: true` operations
- **Status:** ✅ FIXED

---

### BUG-007 · HIGH · Memory / Security
**In-memory nonce cache lost on server restart — replay attack window**

- **File:** `backend/controllers/extension.controller.js` nonce section
- **Root cause:** `nonceCache` is a plain `Map`. On any server restart (which can be triggered by deployment, crash, or Nodemon restart in dev), all nonces are cleared. An attacker who captured a valid request during that window can replay it immediately after restart.
- **Severity:** HIGH — in a single-instance dev/hobby deployment this is acceptable, but should be documented
- **Fix:** Document the limitation prominently and add a NOTE in the code
- **Status:** ✅ DOCUMENTED

---

### BUG-008 · HIGH · Frontend
**`DashboardPage` calls `refreshUser()` in `useEffect` with empty deps — React StrictMode double-invocation**

- **File:** `frontend/src/pages/DashboardPage.tsx` line 33
- **Root cause:** `refreshUser` is a new function reference on every render (defined in AuthContext without `useCallback`). Passing it to `useEffect` deps would cause an infinite loop. Not passing it means the lint warning fires. In StrictMode, the effect fires twice on mount, causing two `/api/auth/me` calls.
- **Fix:** Wrap `refreshUser` in `useCallback` in AuthContext
- **Status:** ✅ FIXED

---

### BUG-009 · HIGH · Code Quality
**`GoogleGenerativeAI` client instantiated on every `analyzeCode` call**

- **File:** `backend/services/aiService.js` line 67
- **Root cause:** `const genAI = new GoogleGenerativeAI(...)` inside the exported function means a new SDK client object is created for every AI analysis request. This wastes memory and loses any connection pooling the SDK provides.
- **Fix:** Hoist to module-level singleton (lazily initialized)
- **Status:** ✅ FIXED

---

### BUG-010 · HIGH · Dead Code / Security
**`dbService.js` contains a hardcoded plaintext password hint and is never used by any controller**

- **File:** `backend/utils/dbService.js`
- **Root cause:** The file was a development scaffold. All controllers use Mongoose directly. The file contains `bcrypt.hashSync('password123', 10)` — a plaintext password that, while hashed, signals the existence of a backdoor dev account.
- **Fix:** Delete the file and remove it from the codebase
- **Status:** ✅ FIXED (file content cleared, controllers unaffected)

---

### BUG-011 · HIGH · Frontend
**`Spinner..tsx` has a double-dot filename — will fail TypeScript path resolution on case-sensitive filesystems (Linux/CI)**

- **File:** `frontend/src/components/Spinner..tsx`
- **Root cause:** Typo in filename. On macOS (case-insensitive) it works; on Linux CI it may fail.
- **Fix:** Rename to `Spinner.tsx`
- **Status:** ✅ FIXED

---

### BUG-012 · HIGH · Auth
**`getMe` response includes the full Mongoose document — exposes internal fields**

- **File:** `backend/controllers/auth.controller.js` line 65 (`getMe`)
- **Root cause:** `res.status(200).json({ data: req.user })` sends the entire Mongoose document. Although `password` is excluded by `.select('-password')`, it still sends `__v`, `syncStatus`, `createdAt`, `updatedAt`, etc., which are not needed by the frontend.
- **Fix:** Select only needed fields in `protect` middleware
- **Status:** 🔶 LOW PRIORITY — no security risk since password is excluded; documented

## MEDIUM

### BUG-013 · MEDIUM · Validation
**`sync.controller` doesn't validate that `leetcodeHandle` / `codeforcesHandle` are safe strings before passing to external APIs**

- **File:** `backend/controllers/sync.controller.js` lines 20, 41
- **Root cause:** A user could store `" ; DROP TABLE "` as their handle. While this can't cause SQL injection (MongoDB), it gets sent directly to the LeetCode/CF GraphQL endpoint.
- **Fix:** Sanitize handles — strip non-alphanumeric characters before external calls
- **Status:** ✅ FIXED

---

### BUG-014 · MEDIUM · Frontend UX
**`AnalysisPage` error handler accesses `e.response?.data?.message` but axios errors are re-thrown as plain `Error` by the interceptor**

- **File:** `frontend/src/pages/AnalysisPage.tsx` line 56
- **Root cause:** The axios interceptor in `axiosInstance.ts` converts all errors to `new Error(message)` — plain Error objects, not AxiosError. So `e.response?.data?.message` is always `undefined`. The correct field is just `e.message`.
- **Fix:** Remove the `.response?.data?.message` chain; just use `e.message`
- **Status:** ✅ FIXED

---

### BUG-015 · MEDIUM · Frontend
**`useProblems` passes `undefined` to `getProblems` when no filters are set**

- **File:** `frontend/src/hooks/useProblems.ts` line 26
- **Root cause:** `getProblems(Object.keys(params).length ? params : undefined)` — passing `undefined` to axios `params` is fine, but it's a latent issue if the API changes. More importantly, the backend `getProblems` controller doesn't handle the case where `platform` filter is passed as an empty string.
- **Fix:** Always pass `params` (even if empty object)
- **Status:** ✅ FIXED

---

### BUG-016 · MEDIUM · Backend
**`triggerAnalysis` in `analysis.controller.js` makes a `problem.save()` call inside an `asyncHandler` but does not handle Mongoose validation errors distinctly**

- **File:** `backend/controllers/analysis.controller.js` line 49
- **Root cause:** If `problem.save()` throws a Mongoose ValidationError, `asyncHandler` wraps it and the error middleware returns it as a 500 with the full Mongoose error message (which can expose schema details).
- **Fix:** Catch Mongoose errors and return a clean 500 message
- **Status:** 🔶 DOCUMENTED

---

### BUG-017 · MEDIUM · UX
**`ProfileInput` calls `saveHandles` and `triggerSync` sequentially without confirming the first succeeded**

- **File:** `frontend/src/components/ProfileInput.tsx` lines 24–26
- **Root cause:** If `saveHandles` throws (e.g. 401), `triggerSync` is still called.
- **Fix:** The `try/catch` already wraps both calls, so if `saveHandles` throws, `triggerSync` is never reached. Actually not a bug — the code is correct. Remove from report.
- **Status:** ✅ NOT A BUG
